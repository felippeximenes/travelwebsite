4. **003** ‚Äî independent of 001/002/004, but touches the same file in nearby regions (`.destination_card`, `.trip_card`); doing it after 002 avoids interleaving unrelated diffs in the same review pass.

## Review notes (post-implementation)

All 7 plans implemented and verified (Playwright: reveal classes, stagger delays, hover-after-reveal specificity, reduced-motion durations, tilt custom-property location, footer `:active` transform ‚Äî all confirmed via computed styles, zero console errors). A manual `review-animations`-standard pass (the skill itself is `disable-model-invocation`, so this was applied by hand against its published standards) found and fixed two coverage gaps introduced by combining plans 003 and 004:

1. **Property-order inconsistency** in `.destination_card`'s base rule vs. its `.show`-scoped and `.trip_card` counterparts (`opacity, box-shadow, transform` vs. `opacity, transform, box-shadow` elsewhere). Harmless functionally (verified), but inconsistent ‚Äî reordered to match.
2. **Reduced-motion coverage gap**: plan 004 only preserved the opacity comprehension-fade on `.hidden`/`.show`; the card-level stagger entrances added by plan 003 (written after 004) still fell through to the blanket 0.01ms override, losing their opacity fade under `prefers-reduced-motion`. Extended the same `transition-duration: 200ms, 0.01ms, 0.01ms !important` treatment to `.destination_grid.show .destination_card` / `.trip_grid.show .trip_card`.

One accepted, non-blocking observation: `.destination_card`/`.trip_card` transition `box-shadow` alongside `transform`/`opacity`. `box-shadow` isn't on the strict transform/opacity-only GPU allowlist, but it doesn't trigger layout (unlike `width`/`height`/`padding`) and this pattern predates this animation pass (present throughout the original redesign). Not worth a site-wide refactor for a handful of small, card-sized elements ‚Äî flagged for awareness, not blocked.
